Module 2 — Structure

HTML
HyperText Markup Language

HTML is the skeleton of the web. Before any styling or interactivity is applied, HTML defines what content exists on a page and what role each piece of content plays — heading, paragraph, navigation, article, or landmark. Getting this right is the foundation everything else depends on.

Overview

What I Learned About HTML

HTML (HyperText Markup Language) was the first technology I encountered in this module, and I initially underestimated it. I assumed it was just about putting text in angle brackets. What I came to understand is that the choices you make in HTML — which element to use, how to nest them, what attributes to include — have real consequences for accessibility, SEO, and how well CSS and JavaScript can interact with the page.

Semantic HTML in particular was a key lesson. Using a <nav> element instead of a generic <div> tells screen readers, search engines, and other developers exactly what that block of content is for.

Document Structure

Anatomy of an HTML Document

Every HTML page follows the same skeletal structure. The DOCTYPE declaration tells the browser which version of HTML to expect. The <head> holds metadata the user never sees directly, and the <body> holds everything they do. 

<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <meta name="description" content="Page description for search engines.">
  <title>My Webpage</title>
  <link rel="stylesheet" href="style.css">
</head>
<body>

  <header>...navigation and branding...</header>

  <main>
    <section>...content...</section>
  </main>

  <footer>...contact and links...</footer>

</body>
</html>

Core Elements

Common HTML Tags

Headings (h1–h6)

Headings define a content hierarchy. There should be only one h1 per page — the main topic — with subheadings nesting beneath it in logical order.

Paragraph <p>

Block-level text content. Browsers add top and bottom margins by default, which should be reset or overridden with CSS rather than using line breaks.

Anchor <a>

Links should always have a descriptive href and, when opening a new tab, include rel="noopener noreferrer" as a security measure.

Image <img>

Always requires a meaningful alt attribute for screen readers. Decorative images should use alt="" so assistive technology skips them.

Form <form>

Forms collect user input. Each input should have a corresponding <label> — connected via for and id — for accessibility and usability.

Lists <ul> / <ol>

Unordered lists for collections without sequence, ordered lists for steps. Never use lists purely for visual indentation — that is a CSS concern.

Best Practice

Semantic HTML Elements

Semantic elements have a built-in meaning that helps browsers, screen readers, and search engines understand your content. Using them correctly is one of the simplest ways to improve a site's accessibility and SEO without extra effort.

🏠
<header>

Introductory content, usually site branding and navigation. Not to be confused with heading elements.

🧭
<nav>

A block of navigation links. Screen readers expose this as a landmark so users can jump to it directly.

📄
<main>

The dominant content of the page. Only one per document. Allows assistive tech to skip repetitive navigation.

📰
<article>

Self-contained content that could be distributed independently — a blog post, a news story, a forum thread.

📦
<section>

A thematic grouping of content, typically with its own heading. Different from a generic div because it implies a topic.

👟
<footer>

Closing content: copyright, contact details, secondary links. Can appear inside an article as well as at the page level.
